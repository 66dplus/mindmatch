# MindMatch — дизайн-спецификация

**Дата**: 2026-05-14
**Контекст**: тестовое задание на позицию автора курса по вайбкодингу (Skyeng)
**Тайм-бокс**: 2:30 чистого времени

---

## 1. Бизнес-контекст

Тестовое предлагает сделать упрощённый Calendly для записи клиентов на консультацию. Мы намеренно идём дальше базового кейса: добавляем **AI-квалификацию клиента внутри формы записи** с автогенерацией **диагностического документа** для клиента и коуча.

**Почему так**: базовый Calendly-клон сделает 80% кандидатов. Наш проект демонстрирует AI не только как инструмент vibe-coding, но и как **продуктовую фичу внутри продукта**, что точно соответствует духу курса.

## 2. Паспорт продукта

| Поле | Значение |
|---|---|
| **JTBD клиента** | «Хочу записаться к коучу личностного роста и не приходить на сессию с пустого листа — пусть коуч уже знает мою ситуацию и подготовит подход» |
| **JTBD коуча** | «Хочу принимать клиентов с подготовленным контекстом, чтобы не тратить первую сессию на сбор анамнеза» |
| **ЦА (выбранная ниша)** | Частные коучи личностного роста и психологи-практики (фрилансер или студия 1-3 чел.) |
| **2 боли ЦА** | (1) Клиенты приходят на первую сессию неподготовленными → 30-40 мин тратится на анамнез вместо работы. (2) Записи теряются в мессенджерах, сложно увидеть свободные слоты. |
| **2 выгоды ЦА** | (1) Клиент приходит с пред-диагностикой → сессия начинается с сути. (2) Все записи в одной таблице, бронирование автоматизировано. |
| **North Star Metric** | **% записей с заполненной AI-квалификацией** (если фича снимает боль клиента — он её проходит; если отпугивает — нет) |
| **3 ключевые фичи MVP** | (1) Запись на свободный слот; (2) AI-квалификация в 5 вопросов внутри формы; (3) Автогенерация диагностического документа для клиента и коуча |
| **НЕ входит в MVP** | OAuth/авторизация, оплата, отмена/перенос, личный кабинет коуча, чат, SMS, синхронизация с Google Calendar, мобильное приложение, push-уведомления |

## 3. Архитектура

```
┌──────────────────────┐
│   Static Frontend    │  React + Vite, hand-coded в Claude Code
│   (Vercel)           │  Single-page wizard
└──────────┬───────────┘
           │ HTTPS
           ↓
┌──────────────────────┐
│  Supabase            │  Postgres + Edge Functions (Deno)
│  - Database          │  RLS отключён для MVP (один коуч)
│  - Edge Function     │  Edge function проксирует OpenRouter
└──────────┬───────────┘
           │ HTTPS
           ↓
┌──────────────────────┐
│  OpenRouter          │  Claude 3.5 Haiku (быстро + дёшево)
│  (LLM gateway)       │  Системный промпт зашит в edge function
└──────────────────────┘
```

### Демо-стек (используется кандидатом):
- **Frontend AI tool**: Claude Code (CLI от Anthropic, в списке тестового)
- **Hosting**: **GitHub Pages** (бесплатно, без SMS-верификации, публичная ссылка `username.github.io/mindmatch`)
- **БД**: Supabase (бесплатный план)
- **LLM**: OpenRouter → Claude 3.5 Haiku

> **Примечание по хостингу**: Vercel в 2026 требует SMS-верификацию, недоступную в РФ. Поэтому берём GitHub Pages — статический хостинг достаточен, т.к. backend живёт в Supabase Edge Functions.

### Альтернативный стек «без VPN» (для урока, как путь для студентов из РФ):
- **Frontend AI tool**: Qwen Coder Agent (chat.qwen.ai, бесплатный, без VPN)
- **Hosting**: GitHub Pages или Cloudflare Pages
- **БД**: Supabase (работает без VPN)
- **LLM**: **GigaChat API** (Сбер, оплата рублями, без VPN) или **Yandex Cloud Foundation Models**

## 4. Схема данных (Supabase)

```sql
-- Типы консультаций
create table services (
  id          bigserial primary key,
  name        text not null,
  duration_min integer not null,
  description text
);

-- Свободные слоты (сидируются на 2 недели вперёд)
create table available_slots (
  id          bigserial primary key,
  service_id  bigint references services(id),
  starts_at   timestamptz not null,
  is_taken    boolean default false
);

-- Записи клиентов
create table bookings (
  id                    bigserial primary key,
  slot_id               bigint references available_slots(id) unique,
  service_id            bigint references services(id),
  client_name           text not null,
  client_email          text not null,
  qualification_answers jsonb,   -- {"q1": "...", "q2": "...", ...}
  diagnostic_report     text,    -- markdown от LLM
  created_at            timestamptz default now()
);
```

**Намеренные упрощения**:
- Нет `users` — коуч в MVP один, входа нет
- `is_taken` обновляется триггером или явно из приложения после INSERT в `bookings`
- RLS отключён — для MVP это допустимо (анонимная запись), но в уроке упомянуть как риск

## 5. Пользовательский флоу

Wizard на одной странице с прогресс-баром:

```
Шаг 1: Hero — описание услуги + CTA "Записаться"
        ↓
Шаг 2: Выбор типа консультации (3 варианта из services)
        ↓
Шаг 3: 5 вопросов AI-квалификации
       1. С каким запросом приходите? (текст)
       2. Что уже пробовали для решения? (текст)
       3. Как поймёте, что сессия прошла успешно? (текст)
       4. Насколько остро сейчас? (слайдер 1-10)
       5. Готовы к домашним заданиям? (да/нет/не уверен)
        ↓
Шаг 4: Выбор слота (сетка дата × время, available_slots WHERE is_taken=false)
        ↓
Шаг 5: Контакты (имя + email)
        ↓
Шаг 6: Loader "Готовим вашу диагностику..." (3-5 сек)
       → POST в edge function → OpenRouter → diagnostic_report
        ↓
Шаг 7: ✅ Подтверждение + диагностика на экране + кнопка "Скачать" (window.print)
```

## 6. AI-промпт (system prompt в edge function)

```
Ты — ассистент коуча личностного роста. На основе ответов клиента
сгенерируй короткий (200-300 слов) диагностический документ из 4 секций:

1. **Суть запроса** (1 абзац, переформулируй своими словами)
2. **Предварительная гипотеза** (что может стоять за запросом — мягко,
   без диагнозов)
3. **Рекомендации к сессии** (3 буллета для коуча: на что обратить внимание)
4. **Что клиенту сделать до встречи** (1-2 простых действия)

Стиль: бережный, на «вы», без медицинских терминов, без оценочных суждений.
Не давай советов уровня «обратитесь к врачу» — это рамка коучинга, не терапии.
```

Модель: **Claude 3.5 Haiku** через OpenRouter — быстро (1-3 сек), дёшево (~$0.0005 за запрос), достаточно качественно для 300 слов структурированного текста.

## 7. Деплой

- **Frontend**: Vite build → GitHub Pages (через `gh-pages` пакет или GitHub Actions). Публичная ссылка: `https://<username>.github.io/mindmatch/`
- **Backend**: Supabase Edge Function → `supabase functions deploy diagnostic`
- **Секреты**: `OPENROUTER_API_KEY` в env-переменных Supabase (никогда не в фронте)
- **CORS**: Edge function отдаёт `Access-Control-Allow-Origin: *` (для MVP допустимо)
- **Vite base path**: в `vite.config.ts` указать `base: '/mindmatch/'` для корректных путей на GitHub Pages

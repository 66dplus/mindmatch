## 1.1. JTBD и паспорт продукта

### Job To Be Done

**Со стороны клиента**:
> Когда я записываюсь к коучу личностного роста впервые, я хочу не «прийти и рассказать всё с нуля», чтобы первая сессия не ушла на сбор анамнеза, а сразу началась с моей реальной темы.

**Со стороны коуча (вторичный JTBD)**:
> Когда ко мне приходит новый клиент, я хочу заранее понимать его запрос, чтобы подготовить подходящий способ работы — а не угадывать на ходу.

### Customer pains & gains

**Боли целевой аудитории — частных коучей личностного роста (фрилансер или студия 1–3 человека):**

1. **Первые 30–40 минут любой первой сессии уходят на «расскажите о себе»**. Клиент платит за час глубокой работы — получает полчаса знакомства. Коуч теряет «теплоту» клиента, который пришёл с энергией обсудить запрос, а вместо этого формулирует биографию.
2. **Записи теряются в мессенджерах**. Клиенты пишут в Telegram/Instagram/WhatsApp, коуч вручную сверяет занятость, копирует данные в гугл-таблицу, отправляет реквизиты. Один пропущенный ответ → потерянная запись.

**Выгоды:**

1. **Сессия начинается с сути**. AI-диагностика дает коучу пред-гипотезу — на первой минуте уже понятно, в каком направлении копать. Клиент чувствует, что его «уже услышали».
2. **Бронирование становится одним сценарием**. Клиент проходит запись, отвечает на квалификацию, видит подтверждение и календарный файл — без переписки.

### North Star Metric

**% записей с полностью заполненной AI-квалификацией**

Почему именно эта метрика:
- Если фича снимает боль клиента, он отвечает охотно — потому что в этом есть смысл лично для него (диагностический документ).
- Если фича отпугивает или ощущается избыточной — клиенты будут бросать форму на середине.
- Альтернативная метрика «% доведённых до оплаты записей» зависит от слишком многих переменных (цена, ниша, доверие к коучу) и не показывает ценность именно этой фичи.

Прокси-метрики, которые мы будем смотреть в первый месяц:
- Conversion от «открыл сайт» к «нажал записаться»
- Conversion от «начал квалификацию» к «дошёл до подтверждения»
- Средняя длина ответов на текстовые вопросы (показатель вовлечённости)

### Паспорт продукта

| | |
|---|---|
| **Цель** | Дать коучу личностного роста инструмент, который заменяет переписку «запись + анамнез» одним коротким сценарием со встроенной AI-диагностикой запроса. |
| **Целевая аудитория** | Частные коучи и психологи-практики, работающие соло или в студии до 3 человек. Не корпоративные программы. Не терапевты с медицинской лицензией. |
| **3 ключевые функции MVP** | 1. Запись на свободный слот из расписания коуча. 2. AI-квалификация — 5 вопросов о запросе внутри формы записи. 3. Автогенерация диагностического документа для клиента и коуча по итогам ответов. |
| **Что НЕ входит в MVP** | Авторизация коуча; оплата онлайн; отмена/перенос записи; личный кабинет коуча; уведомления в Telegram/SMS; интеграция с Google Calendar; мобильное приложение; чат; multi-coach маркетплейс; видеосвязь внутри платформы. |
| **Длительность MVP-разработки** | ~2:30 чистого времени через vibe-coding (Claude Code + Supabase + GitHub Pages). |

## 1.2. Архитектура и выбор инструментов

### Архитектура

```
┌──────────────────────┐
│   Static SPA         │  React + Vite — single-page wizard
│   (GitHub Pages)     │  https://66dplus.github.io/mindmatch/
└──────────┬───────────┘
           │ HTTPS
           ↓
┌──────────────────────┐
│  Supabase            │  Postgres (services, slots, bookings)
│  - Database          │  + Edge Function (Deno) на TS
│  - Edge Function     │  RLS включён, anon insert разрешён для MVP
└──────────┬───────────┘
           │ HTTPS (server → server)
           ↓
┌──────────────────────┐
│  OpenRouter          │  Шлюз к Claude 3.5 Haiku.
│                      │  Системный промпт зашит в Edge Function.
└──────────────────────┘
```

### Основной стек (демо-реализация)

| Слой | Инструмент | Почему именно он |
|---|---|---|
| Vibe-coding-инструмент | **Claude Code** (CLI от Anthropic) | Прямо упомянут в задании рядом с Cursor/Replit. Даёт полный контроль над структурой, не «угадывает» как Lovable, можно вести через короткие итеративные промпты. |
| Frontend framework | **React + Vite** | Vite — самый быстрый dev-server для статических SPA. React всё ещё де-факто стандарт для AI-генерации компонентов. Бандл 211 КБ uncompressed → влезает в free-план GitHub Pages с запасом. |
| Стилизация | **Чистый CSS + CSS-переменные** | Без Tailwind — для урока проще объяснить «вот файл с цветами, вот стили». Variable fonts (Fraunces) даёт богатую типографику без npm-пакетов. |
| БД и backend | **Supabase** (Postgres + Edge Functions) | Бесплатный план с реалистичными лимитами (500 МБ БД, 500K вызовов функций/мес). Postgres даёт настоящие реляционные связи и RLS. Edge Functions на Deno — TS из коробки. |
| LLM-провайдер | **OpenRouter → Claude 3.5 Haiku** | OpenRouter аккаунт оплачивается криптой (без VPN), даёт доступ к Claude/GPT/Llama по одному ключу. Haiku — оптимум по соотношению цена / скорость / качество для 300-словного структурированного текста (~1 цент за 100 диагностик). |
| Деплой | **GitHub Pages + GitHub Actions** | Vercel/Netlify в 2026 требуют SMS-верификацию, недоступную в РФ. GitHub Pages — бесплатно, без верификации, открытый репо = бонус к оценке. CI/CD в Actions. |

### Альтернативный стек для студентов из РФ без VPN

Все вышеперечисленные инструменты имеют альтернативы, доступные без VPN:

| Слой | Альтернатива | Комментарий |
|---|---|---|
| Vibe-coding-инструмент | **Qwen Coder Agent** (chat.qwen.ai) | Бесплатный chat-агент с инструментами от Alibaba. Работает без VPN. Чуть менее «структурный», чем Claude Code — компенсируется более явными промптами. |
| Frontend / Backend / Хостинг | React + Vite + Supabase + GitHub Pages | Все четыре доступны из РФ без VPN — Cloudflare-фронтенд для GitHub и AWS-инфра Supabase достижимы. |
| LLM | **GigaChat API** (Сбер) | Оплата рублями, без VPN. API похож на OpenAI Chat Completions — нужно только заменить URL и заголовки. Чуть менее точен в литературном тоне, но достаточен для 300-словного отчёта. |
| LLM (альтернатива №2) | **Yandex Cloud Foundation Models** | Хостит open-source модели (Llama, Yandex GPT). Биллинг в рублях. |

Полный путь «студент из РФ без VPN»: Qwen Coder Agent → React + Vite + Supabase + GitHub Pages → GigaChat. Все шаги проходимы без обходных средств; в уроке это разобрано отдельным пунктом.

### Схема данных

```sql
-- services: типы консультаций
services(
  id           bigserial PK,
  name         text,                  -- "Карьерный фокус"
  duration_min integer,               -- 60
  description  text
)

-- available_slots: свободные слоты коуча (сидируются на 2 недели вперёд)
available_slots(
  id         bigserial PK,
  service_id bigint FK → services.id,
  starts_at  timestamptz,
  is_taken   boolean default false,
  UNIQUE (service_id, starts_at)
)

-- bookings: подтверждённые записи + ответы клиента + AI-диагностика
bookings(
  id                    bigserial PK,
  slot_id               bigint FK → available_slots.id,
  service_id            bigint FK → services.id,
  client_name           text,
  client_email          text,
  qualification_answers jsonb,    -- {q1: "...", q2: "...", q3: "...", q4: 7, q5: "Да"}
  diagnostic_report     text,     -- markdown от LLM
  created_at            timestamptz default now()
)
```

**Намеренные упрощения и почему:**

- **Нет таблицы `users`**. В MVP коуч ровно один; авторизация не нужна. В продакшне был бы `users` с `auth.uid()` и `coach_id` в services/slots.
- **`is_taken` обновляется триггером** `mark_slot_taken()` после INSERT в bookings — на уровне БД, не приложения. Это исключает гонки.
- **RLS включён, но anon-инсерты разрешены** в `bookings`. Это правильный паттерн для публичной формы записи: anon может ВСТАВИТЬ, но не может ЧИТАТЬ чужие записи. Service role читает всё для коуча.
- **`qualification_answers` как jsonb**, а не отдельные колонки. Для MVP — гибкость > валидации. Если структура вопросов изменится, не нужна миграция.

### Связи (ER-диаграмма)

```
services (1) ────< (N) available_slots
   │                       │
   │                       │ (1)
   │                       ↓
   └──────< (N) bookings >─┘
                  │
                  └─ qualification_answers (jsonb)
                  └─ diagnostic_report (text, AI-generated)
```

## Приложение А. Чем работа сильнее «обычного Calendly-клона»

Базовое решение из задания — Calendly-like форма + сохранение в Sheets/Supabase. Это сделает большая часть кандидатов.

Наш подход добавляет одну фичу — **встроенную AI-квалификацию с диагностическим документом** — и за счёт этого работа усиливается по всем критериям оценки:

| Критерий теста | Вес | Что обычно показывает кандидат | Что показано здесь |
|---|---|---|---|
| Продуктовое проектирование | 20% | JTBD «записать клиента» | Уникальный JTBD «прийти на сессию подготовленным» + продуктово-осмысленный North Star |
| Vibe-coding | 25% | UI + CRUD-операции | UI + CRUD + AI как **продуктовая фича внутри продукта**, не только как инструмент кодинга |
| Архитектура и no-VPN альтернатива | 15% | Один стек | Параллельный no-VPN путь, проработанный до уровня LLM-провайдеров |
| Деплой | 15% | Vercel-ссылка | GitHub Pages с открытым репо (можно посмотреть код), CI/CD в Actions |
| Prompt engineering | 10% | Промпты к UI-генератору | Промпты к Claude Code + **боевой system-prompt внутри edge function** |
| Образовательный фрагмент | 15% | «Как сделать форму» | «Как встроить AI-фичу в продукт» — более релевантная тема для курса по вайбкодингу |

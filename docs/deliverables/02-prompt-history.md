# История промптов

**Проект**: MindMatch
**Инструмент**: Claude Code (CLI от Anthropic)
**Дата**: 14 мая 2026

Ниже — настоящая последовательность ключевых промптов, которые привели от пустой папки к задеплоенному MVP. Между промптами я не правил код руками — все правки делались следующим промптом.

---

## Промпт 1. Анализ ТЗ и предложение продуктовой идеи

> «У меня есть тестовое `@Тестовое_задание_для_автора_курса_по_вайбкодингу.docx`. Хочу чтобы ты внимательно ознакомился и выполнил так, чтобы меня взяли на эту позицию. Моя идея: запись в календарь + AI-квалификация, где мы предлагаем клиенту пройти ИИ-квалификацию и выдаём ему диагностический документ. /brainstorming»

**Что вернул AI**: разобрал ТЗ по критериям оценки, оценил мою идею, обозначил риск (over-engineering за 2-3 часа), задал 4 уточняющих вопроса о нише, БД, тайминге AI-квалификации и стеке.

**Чему научило**: даже хорошую идею стоит «продать» AI вместе с контекстом задачи. Без `/brainstorming` и без приложенного ТЗ AI начал бы кодить раньше времени.

---

## Промпт 2. Спор о технических решениях

> «Давай возьмём SQLite. Это будет просто.»

**Что вернул AI**: вместо «ок, делаем» — структурированный pushback. Объяснил, что SQLite ломается на serverless-хостингах (Vercel, Netlify эфемерны), предложил Supabase с обоснованием «легче объяснить в уроке студенту».

**Чему научило**: правильно настроенный AI-ассистент **должен спорить**, когда видит технический риск. Если он соглашается с любым решением — это плохой ассистент. Я в итоге выбрал Supabase именно по тому аргументу, что его легче преподавать.

---

## Промпт 3. Этическая развилка

> «Давай ты сделаешь всё, а напишем в документе что использовали Qwen Coder. Потому что Lovable, Bolt, Stitch не работают без VPN.»

**Что вернул AI**: отказ + объяснение почему. Перечитал ТЗ внимательно, нашёл, что требование «без VPN» относится к УРОКУ для студентов, а не к моему демо. Предложил честный вариант: «Демо честно делаем со мной (Claude Code), в уроке для no-VPN сценария описываем Qwen Coder + GigaChat».

**Чему научило**: AI с правильными инструкциями отказывается от компромиссов, ведущих к лжи в сдаваемых артефактах. Это сохраняет и репутацию, и оффер.

---

## Промпт 4. Дизайн темы фронтенда

> «Делай весь фронт через `frontend-design` skill. Сервис для коучей личностного роста — нужен бережный, литературный тон, не корпорейт и не "велнес".»

**Что вернул AI**: выбрал editorial-organic aesthetic — Fraunces (variable serif) для display + DM Sans для body, тёплая бумажная палитра (`#F4EFE6` / `#1F1B16`), акцент saffron (`#D67340`), sage (`#6B7A65`). Добавил субтильную grain-текстуру на фоне через layered CSS gradients.

**Что я уточнил следующим промптом**: «Сделай орнамент в правом верхнем углу hero — концентрические круги тонкой линией, как символ слоёв запроса.» → получилось.

**Чему научило**: визуальный стиль лучше задавать через **аналогии и эмоции** («бережный литературный журнал», «не корпорейт»), чем через цвета/шрифты. AI делает лучший выбор палитры, чем я.

## Промпт 5. Backend и интеграция

> «Напиши Supabase Edge Function на TypeScript / Deno, которая принимает {service, answers, clientName}, дёргает OpenRouter с системным промптом для диагностики, возвращает markdown. CORS — `*` для MVP. Модель — Claude 3.5 Haiku.»

**Что вернул AI**: 80 строк кода с обработкой OPTIONS-preflight, шаблонизированным системным промптом (4 секции диагностики), `User-Prompt`-конструктором из ответов клиента, аккуратной обработкой ошибок OpenRouter и fallback на 502 при сетевых сбоях.

**Уточняющий промпт**: «Добавь `HTTP-Referer` и `X-Title` в заголовки OpenRouter — иначе плохая трассировка в их dashboard.» → добавлено.

**Чему научило**: для backend-кода AI лучше работает с **точно сформулированным контрактом** (что приходит, что возвращается, какие ошибки). Без контракта он генерирует «универсальный» код, который потом неудобно вызывать.

## Промпт 6. Деплой и автоматизация

> «Деплой не на Vercel (требует SMS, не работает в РФ). Сделай GitHub Pages с GitHub Actions: на push в main собирает Vite, выкатывает на pages. Env-переменные подставляются из GitHub secrets.»

**Что вернул AI**: workflow `.github/workflows/deploy.yml` на 50 строк с двумя джобами (build + deploy), правильными permissions (`pages: write`, `id-token: write`), кэшированием npm и подключением Vite env-переменных через `${{ secrets.* }}`.

**Чему научило**: для деплоя проще описать ОГРАНИЧЕНИЕ («не Vercel, причина X») и ЦЕЛЬ («GitHub Pages с CI/CD»), чем расписывать каждую строку YAML. AI сам знает GitHub Pages best-practices.

## Эволюция: что не сработало и как я доправлял

### Итерация 1 → 2 (форма квалификации)

**Первый вариант от AI**: 5 вопросов на отдельных шагах wizard (один вопрос — один экран).

**Моя правка**: «Сделай 5 вопросов на одном экране со скроллом. Психологически проще: клиент видит "всего 5 вопросов", а не "сколько ещё впереди".» → AI переработал в `q-list` с одним финальным CTA «Дальше — выбрать время».

### Итерация 2 → 3 (mock-режим)

**Первая версия**: приложение падало без env-переменных Supabase.

**Моя правка**: «Сделай так, чтобы без `VITE_SUPABASE_URL` приложение работало на mock-фолбэке: услуги из локального файла, диагностика через локальный stub. UX должен пройти end-to-end даже без бэкенда.» → AI добавил `hasSupabase` флаг и `mockDiagnostic()`-fallback. Это критично для демо — деплой работает с первого пуша, без настройки секретов.

### Итерация 3 → 4 (диагностика как продукт, не как отчёт)

**Первая версия системного промпта**: 5 секций, технический тон, упоминания «пациента».

**Моя правка**: «Перепиши system prompt. Это рамка КОУЧИНГА, не терапии. Никаких слов "диагноз", "пациент", "обратитесь к врачу". Тон бережный, на "вы", без оценочных суждений. 4 секции, не 5.» → AI переработал, теперь промпт явно ограничивает себя коучинговой рамкой.

## Метаурок

Промпты к AI — это не вопросы, а **контракты**. Хороший промпт содержит:

1. **Цель** («что я хочу получить»)
2. **Контекст** («где это будет жить, кто это читает»)
3. **Ограничения** («что НЕЛЬЗЯ делать»)
4. **Стиль / тон** для UI и LLM-фичей

AI обрабатывает все четыре элемента одновременно. Чем явнее каждый — тем меньше итераций.

И главное: AI должен **спорить**. Если он соглашается со всем, я делаю что-то не так — либо мой промпт не оставляет места для альтернатив, либо я не настроил AI на pushback. В правильном диалоге AI несколько раз сказал «не делайте этого вот так, потому что…» — и это сэкономило несколько часов.
